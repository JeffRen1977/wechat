# 一篇看懂 AI Agent 三大核心

## 这支视频在讲什么
- **主角：**白白说大模型最新视频，将 AI Agent 的「操作系统」拆成三块：Function Calling、Model Context Protocol（MCP）与 Agent-to-Agent（A2A）。
- **核心问题：**为什么单一大模型越来越难满足复杂任务？因为链路长、上下文碎片，必须靠标准化接口与协同机制来补齐。
- **作者立场：**视频更像实战复盘——不是抽象概念，而是告诉你“怎样把 Agent 接入真实世界的数据与工具”。

## 三大核心技术的本质
### Function Calling：让模型变成「指挥官」
- **要点：**Function Calling 是把外部函数的签名暴露给模型，让它能在回答前主动挑选、填参并调用。例如“查库存”“发起支付”，模型只需决定何时调用，真正的执行仍由后端服务完成。
- **实操建议：**
  - **函数描述越口语化越好**，模型才能选对工具。
  - **幂等+重试**是底线，避免重复扣款、重复建单。
  - **输入输出要缩写**，记得在函数 schema 里声明字段含义，别把整份文档塞进 prompt。

### MCP：把上下游信息装进「共享内存」
- **场景：**当 Agent 需要读写企业知识库、数据库或第三方 API 时，MCP（Model Context Protocol）提供统一的资源抽象，让模型以「文件系统」「工具」「事件流」的方式拉取上下文。
- **好处：**
  - **分层权限**：不同角色 Agent 只看到自己该看的资源。
  - **上下文去重**：用引用（resource pointer）代替整段贴 paste，省 token 也减少幻觉。
  - **实时性**：可订阅外部事件，触发后续 Function Calling。

### A2A：Agent 之间的团队协作
- **定义：**Agent-to-Agent 不是“模型加模型”，而是通过约定好的接口，把不同能力的 Agent 串成工作流。例如：调研 Agent 负责拉资料，决策 Agent 负责综合，执行 Agent 负责 Function Calling。
- **关键机制：**
  - **共识消息格式**：推荐使用 JSON schema 或对话模板，确保 baton 交接清晰。
  - **任务分派策略**：可以是规则引擎（if-else）也可以是调度模型，但务必记录日志，方便回溯。
  - **超时与仲裁**：若某个 Agent 长时间无响应，必须有守门员接管，避免“无限等待”。

## 三者如何组合成一条「链路」
1. **入口 Agent** 拿到需求后，先调用 MCP 读取必要的背景（如客户画像、项目历史）。
2. **分析 Agent** 基于这些上下文拆解子任务，决定哪些需要 Function Calling（如调用 CRM、生成报价）。
3. **执行 Agent** 负责实际 API/工具调用，并把结果写回 MCP 共享区。
4. **复核 Agent**（A2A 的一环）检查输出，必要时再次触发函数或请求人工介入。

这种流水线的好处是：
- **上下文留痕**：每一步都沉淀在 MCP，方便监管与学习。
- **容错**：任一节点失败，调度器都能重跑该节点，而不是重来整次对话。
- **可扩展**：要接新工具，只需在 Function Calling 层增加签名；要加新角色，就在 A2A 配置里添加并行节点。

## 适合谁、怎么落地
- **中小团队**：先从 Function Calling 入手，把最痛的 2–3 个流程（如内部 FAQ、报表生成）模块化；MCP 可以用轻量对象存储或向量库模拟。
- **有合规要求的行业**：优先部署 MCP，借助权限与审计功能满足内控，再逐步上 A2A 自动化。
- **To C 产品**：别一开始就搞多 Agent，先让 Function Calling + MCP 打通数据，再考虑在客服场景中引入 A2A 做分层回复。

## 常见坑位 Checklist
- **函数粒度过大**：一次调用包办所有逻辑，模型看不到中间状态，调试困难。
- **MCP 权限混乱**：没有把读写分开，导致 Agent 无意间覆盖生产数据。
- **A2A 没有“收官”角色**：多个 Agent 来回扯皮，最终没人对产出负责。
- **Metrics 缺失**：至少追踪“调用次数、成功率、平均耗时”，否则无法定位瓶颈。

> 一句话记住：Function Calling 解锁工具手、MCP 是共享脑海，A2A 则让 Agent 团队化。三者组合，才能把大模型从「会聊天」推进到「能交付」。
