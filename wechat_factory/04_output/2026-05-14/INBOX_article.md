# 工具调用要快稳，这5款小模型就够了

开源圈这两周最常见的吐槽是：「想让 Agent 跑通工具链，光是推一个大模型就把显卡烤红了。」其实，工具调度的难点不在算力，而在稳定地格式化参数、串联上下游。好消息是，越来越多小模型已经补齐这块短板。

## 算力焦虑：大模型不是每个团队都扛得起

最近的项目复盘里，最常见的失败原因就是 **延迟不可控**。前台一个 30 秒的空白旋转圈，后台可能是 70B 大模型在重算 JSON。用户体感差、成本又高。

更现实的是，很多内部工具只需要 **结构化调用和容错**，并不需要动辄数十万上下文的推理。以参考文章提到的 SmolLM3、Qwen3 等模型为例，它们把重点放在「接受工具 schema → 输出合法 JSON」这件事情上，算力预算直线下降。

> 你要的不是「最聪明」，而是「最守规矩」。能老老实实给出你要的字段，就是合格的工具模型。

## 5 款小模型的工具调用亮点

💡 **SmolLM3-3B**：64K 长上下文、双模式推理、原生 JSON/XML 与 Python 函数双接口。对已有 Python 工具链的团队友好，用 xml_tools 接企业内部旧系统尤其省心。

💡 **Qwen3-4B-Instruct-2507**：原生长上下文 262K，支持 100 多种语言，结合 Qwen-Agent 模板能直接把 MCP 工具挂进去。对于跨语种客服、跨区业务是一大加分项。

💡 **Phi-3-mini-4k-instruct**：体积小，4K 上下文，微软官方对 Transformers 的 chat template 做了适配，想要快速在 Hugging Face 推理空间里验证工具调用的，可以直接套官方示例。

![配图](wechat_factory/05_assets/images/2026-05-14_INBOX_fig1.png)

💡 **LLaMA Guard 3 1B**：别把它只当审核模型。它能触发「阻断」「升级人工审核」等安全工具，等于给 Agent 系统加了一个判官层，特别适合需要合规日志的金融、医疗团队。

💡 **Mistral-Nemo-Instruct-2407**：12B 虽在「小模型」上限，但胜在 **JSON schema 自检**，能在一次对话内纠正格式错误。需要落地在单张专业 GPU 上的企业可以考虑 INT4 量化部署。

## 如何把小模型接进自己的 Agent 流水线

1. **统一工具协议**：不管你最终选谁，先把工具的输入输出 schema 固定下来，最好直接用 JSON Schema 或 Pydantic 约束。这样换模型时只改提示词，不动底层代码。
2. **提示词里写清「出错自救」规则**：例如「若解析失败请主动重试一次」。SmolLM3、Mistral-Nemo 都能按照指令自检，省掉大量 try/except。
3. **分层路由**：把「闲聊/创意」路由到大模型，把「确定性工具调用」路由到小模型。很多团队的体验就是 —— 95% 的请求其实都能在小模型层解决。

## 哪些场景仍该保留大模型？

⚠️ 超长链路推理、需要公开答案解释权、或一问多工具的复杂 orchestration，依旧建议保留大模型兜底。尤其涉及监管披露或关键业务指标时，小模型给出的答案需要再与大模型交叉比对。

---

> 预算有限但又想让 Agent 可靠执行指令的团队，可以先用这 5 款小模型排个队，对照自己的数据敏感级别与国际化程度去筛选。

### Action Items

1. ✅ **列出你现有工具的输入输出 schema**，检查是否能直接映射到 JSON，不行的话先做字段梳理。
2. ✅ **挑 2 款小模型做 A/B 测试**，分别验证「格式合规率」与「延迟」，测试脚本保持一致。
3. ✅ **建立路由 + 监控仪表盘**，实时记录是哪一类请求触发回退到大模型，方便后续优化提示词或工具。
