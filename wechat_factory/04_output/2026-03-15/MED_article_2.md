# 想把 3D 医学大模型塞进诊室？先让 MedPruner 把 95% 冗余 token 删掉

## 研究背景：3D 医学 VLM 为什么总是“卡脖子”
近两年多模态大模型攻入医学影像，Med-PaLM M、MedGemma 这样的 VLM 能同时读 CT、MRI 与报告。但一旦面对 3D 体数据，就会遭遇两个现实问题：其一是将数百张切片直接拼接造成 token 爆炸，推理速度慢到无法贴近临床；其二是大量背景切片信息密度低，却挤占了上下文窗口。来自中山大学、浙大和谷歌医学团队的最新工作 MedPruner（arXiv:2603.11625，2026.03.12）给出一个训练-free、模型无关的解决方案，目标是在不牺牲诊断性能的前提下，把视觉 token 压到原来的 5% 以内。

## 方法速览：分两步削掉“无效切片 + 冗余 token”
- **阶段一：切片锚点筛查（Inter-slice Anchor-based Filtering）**。先对 3D 体数据做切片层面的信息密度估计，直接滤掉那些高度相似、几乎无病灶的层面，避免把噪声送入 LLM。
- **阶段二：动态信息核选择（Dynamic Information Nucleus Selection）**。根据视觉 backbone 的注意力累计曲线，自适应决定 token 保留阈值；病灶边界、纹理等高权重区域会保留下来，背景 token 则被剔除。
- **完全无须再训练**：对 Hulu-Med、MedGemma、Qwen3-VL 等不同体积的 VLM 都是 plug-and-play，兼容多种 3D 医学基准（3DRad、M3D、AMOS-MM 等）。

## 实验结果：2.46% token 仍能拿下满分
- **大幅压缩仍保性能**：在 AMOS-MM 数据集上，MedPruner 让 MedGemma 仅保留 2.46% token，平均指标反而提升到原模型的 100.65%，推理时间从 38.0 秒降到 35.9 秒。
- **在多切片场景胜过原模型**：面对平均 87 层、最多 600+ 层的 M3D VQA，Hulu-Med 搭配 MedPruner 在 BLEU-4（12.580）与 ROUGE-1 上都超越未压缩基线，同时把 token 保留率降到约 52%。
- **跨模型一致生效**：无论是 7B 模型（Hulu-Med）、4B 模型（MedGemma）还是 8B 的 Qwen3-VL，都能获得 40%–55% 的 token 削减、0.2–1.3 秒的加速，而指标波动控制在 2% 内。
- **注意力分布更健康**：论文指出 MedGemma 的注意力高度集中在少量位置，MedPruner 动态调整阈值后，可自动聚焦病灶边界等关键区域，避免上游 backbone 的“平均用力”。

## 临床价值：把 3D 多模态模型的“体重”减下来
1. **满足实时交互**：将单例处理速度从 9.2s 压到 7.9s（Hulu-Med），为 PACS、术中导航等场景释放算力预算。
2. **降低部署门槛**：医院侧 GPU 普遍紧缺，训练-free 的剪枝框架可以直接集成在推理代码里，无需再 fine-tune。
3. **更易做多病种场景**：当 VLM 需要同时读取动态增强 CT、MRI 等多序列时，MedPruner 可以在不同模态共享同一套过滤逻辑，避免管线爆炸。

## 上线前还要补的课
- **路径依赖**：当前剪枝策略依赖视觉 backbone 的注意力质量，如果上游模型本身对噪声敏感，MedPruner 也会跟着失准。
- **缺少任务级指标**：实验主要用 ROUGE/BLEU/ METEOR，临床更在意病灶检出率或病人级诊断一致性，需要额外验证。
- **部署安全**：极端情况下 97% token 被剪掉，需配合不确定性评估与兜底策略，避免病灶遗漏。

## 我们怎么看
MedPruner 给出了一个“无需换模型，就能啃 3D 医疗数据”的落地样板。它把压缩问题拆成“切片级 + token 级”两个维度，兼顾了训练成本与推理效率。对国内影像中心而言，该方案可以嵌进现有 MedVQA 或报告生成系统，直接减轻算力压力。下一步值得关注的是：能否进一步结合多模态数据（影像 + EHR 文本）做联合剪枝，以及如何把 token 削减带来的显存空余，用于部署更高分辨率或更多病种的提示词。换句话说，MedPruner 打开的是“让医学大模型轻装上阵”的一条现实路径。
