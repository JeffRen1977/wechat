# 760 个金融 API，谁敢接？FinToolBench 给 LLM Agent 来了场真刀真枪的测验

## 背景：金融场景需要“可追溯”的工具链
在投研、合规、客服等场景，LLM 不再只是写文案，而要调用行情、基金、监管接口，且每一步都可复盘。现有评测要么停留在文档 QA，要么依赖玩具 API，难以衡量真实工具使用的可行性。上海 AI Lab 团队推出 FinToolBench（arXiv:2603.08262，2026.03.09）：一个包含 760 个可执行金融工具、295 道必用工具问题的基准，重点考察“调用率、执行成功率、合规匹配度”三大维度。

## 设计亮点：不仅要会调，还要调得合规
- **金融属性标签**：每个 API 都被标记 timeliness、Intent（资讯/交易）与监管域，模型调用不合规就记入 TMR、IMR、DMR 三类 mismatch。
- **执行数据闭环**：统一的工具调用接口 + JSON Schema，所有调用会记录参数、返回值及错误，便于审计。
- **评估指标**：
  - **TIR**（Tool Invocation Rate）：模型愿不愿意动手。
  - **TESR**（Tool Execution Success Rate）：调用后是否得到有效输出。
  - **CER = TESR / TIR**：在“已经动手”的情况下成功率有多高。
  - **Soft Score/CSS**：答案正确度与执行链条质量的综合分。

## 实测表现：激进与保守各有短板
- **Qwen3-8B**：TIR 高达 0.8712，Soft Score 0.4234 全场最高，但 CER 只有 0.3385，说明“敢调但常报错”。
- **Doubao-Seed-1.6**：TIR 0.6508、TESR 0.3254、CER 0.5000，执行最稳健，但 TMR 0.3438、IMR 0.6563，表明在意图/监管匹配上仍易翻车。
- **GPT-4o**：最保守，TIR 只有 0.2267，TESR 0.1400；不过一旦出手，CER 0.6176、CSS 0.6700、IMR 0.5、DMR 0.1176，为全场最低 mismatch。它宁可不回答，也不冒险。
- **GLM-4.7-Flash**：TIR 0.4407、TESR 0.2102，且三类 mismatch 都偏高（TMR 0.4615），暴露了参数解析与监管意识薄弱的问题。

## 为什么要加 Finance Attribute Injection？
团队还做了“有无金融标签提示”的对比：加入 timeliness/Intent/Domain 说明后，TIR 略降，但 CER 与 CSS 明显提升，IMR、DMR 均下降。这意味着显式提示反而能减少“为了凑答案硬调工具”的情况，让链路更可追溯。

## 对落地团队的启示
1. **评估工具链，不止评估模型**：FinToolBench 用统一执行环境控制变量，帮助你判定问题出在 planner 还是 tool wrapper。
2. **要有合规指标**：单看准确率无法发现“越界”调用，TMR/IMR/DMR 的思路值得借鉴到实际监控里。
3. **策略要分层**：调度器需根据 timeliness、权限级别决定是否出手，GPT-4o 的“稳”与 Qwen 的“敢”之间还有大量平衡空间。

## 我们怎么看
FinToolBench 让“金融 Agent”首次拥有接近真实的跑道：从基金分析、宏观解释、券商合规到监管问答，都需要调用真实 API，并留下可审计 trace。国内券商、银行可在沙箱环境中复刻这套评测流程，对内部 Agent 做“工具能力体检”。下一步值得关注的是：如何将 FinToolBench 扩展到中文数据源与本地监管 API，以及围绕 TIR/CER 等指标做动态调度策略，让 Agent 在“敢用工具”与“用得合规”之间找到更优解。
