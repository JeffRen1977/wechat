# FinRetrieval：别再只刷榜，先搞定“查一行财报数据”这件小事

## 数据集速览
- Stanford 团队发布 FinRetrieval（arXiv:2603.04403，2026.01.02）：500 道“提取具体财务数值”的检索题，覆盖 14 种 AI Agent 配置、3 家厂商（Anthropic/Google/OpenAI）以及所有工具调用日志。
- 目标：评测大模型代理在真实 API + WebSearch 环境下拉取财务数据的能力，而非只在静态问答里背答案。

## 关键发现
1. **工具决定一切**：Claude Opus 配备 MCP 财务 API 时准确率 90.8%；仅靠 WebSearch 就跌到 19.8%，**差 71 个百分点**。其他模型差距顶多 21pp。
2. **推理只是锦上添花**：在有 MCP 的情况下，OpenAI 推理模式能多 9pp，Claude +2.8pp，Gemini +4pp；但没工具时，推理再好也无济于事。
3. **区域差异来自财年命名**：US 公司 vs 非 US 公司准确率差距 5.6pp（Claude）。原因不是模型看不懂外语，而是 API 以“财年起始年”命名，模型默认“终止年”，导致日本/印度非 12 月财年的查询全错。
4. **延迟成本**：推理模式让 OpenAI 延迟从 9.9s 跳到 31.2s（3.2×）；Claude/Gemini 增幅 ~1.2×，说明不同供应商在“深思模式”下的时延开销差异巨大。

## 业务启示
- **集成优先级**：对投研 Agent 而言，比起追逐最新模型版本，不如先把官方财报 API、10-K 解析端点集成到位，收益比升版本更大。
- **文档/接口标准化**：FinRetrieval 的地理差距几乎完全由“财报命名约定”导致——企业内部若也存在不同 fiscal 命名，需提前提供映射或统一约定。
- **推理策略按需开**：OpenAI 推理模式虽然提升 9pp，但代价是 3× 延迟；对需要即时响应的业务，可考虑只对难题启用。
- **公开数据助力调参**：该基准发布了全部工具调用 trace，方便团队复现实验、调策略、甚至构建自家“检索能力”对比。

## 我们建议
1. 建立“工具健康度”监控：对常用财务端点做 SLA 监控，任何失败要回落到备用数据源，避免准确率瞬间跌 70pp。
2. 设计“区域补丁”策略：针对非 12 月财年公司，在 Query 构造阶段自动补齐起始/终止年、季度映射。
3. 把 FinRetrieval 纳入模型选型流程：用它作为“工具利用能力”的强制测试，比只跑静态问答更能看出代理能否真查到数字。
